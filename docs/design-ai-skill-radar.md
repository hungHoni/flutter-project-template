## Problem Statement

Keeping up with AI is overwhelming. News, tools, frameworks, and techniques change weekly across Reddit, HN, Twitter, blogs, and newsletters. A software engineer who wants to stay current has to check 5+ scattered sources daily, with no way to know what's actually relevant to their specific role and existing skills. The result: either information overload or falling behind silently.

## What Makes This Cool

This isn't another RSS reader or AI newsletter. It's a **personal skill diff engine**: it knows what you already know, watches what's trending, and tells you exactly what's new and relevant to *you* — with a concrete action to close each gap. The "3 new gaps detected this week" push notification is the hook: curiosity-driven, personalized, actionable.

## Constraints

- Flutter mobile app (iOS + Android)
- Minimalist editorial UI with Anduin blue (`#3FA6EE`) as the primary accent color
- Claude API for summarization and skill extraction
- RSS-based feed ingestion (Reddit, HN, blogs) — no scraping/auth required for MVP
- Solo developer building for personal use first
- Use **Stitch MCP** for UI component development and screen generation
- Follow the minimalist-flutter-ui design system: serif display, sans-serif body, monospace meta, flat components, warm monochrome base palette + Anduin blue accents

## Premises

1. The MVP starts with feed aggregation (RSS) — without good signal input, the AI analysis has nothing to work with.
2. Flutter mobile app is the right platform — the user wants to check this on their phone.
3. Claude API is the right tool for summarization and skill extraction — keyword matching can't understand context or generate actionable briefs.
4. Personalization (knowing your role + skills) is the key differentiator — without it, this is just another RSS reader.

## Approaches Considered

### Approach A: Feed-First MVP
- **Summary:** Build the RSS aggregator + minimalist reader first. Reddit, HN, and blog RSS feeds displayed in a clean editorial UI. No AI, no personalization yet.
- **Effort:** S (human: ~3 days / CC: ~30 min)
- **Risk:** Low
- **Pros:** Immediately usable, learn Flutter fast, solid foundation to layer AI on
- **Cons:** It's "just" an RSS reader until you add the AI layer
- **Completeness:** 4/10

### Approach B: Thin Vertical Slice
- **Summary:** Onboarding (role + skills) → RSS feed ingestion → Claude API summarization + gap detection → dashboard showing briefs + gaps. End-to-end but narrow (3-5 sources, one role).
- **Effort:** M (human: ~2 weeks / CC: ~2 hours)
- **Risk:** Medium (API costs, scraping reliability)
- **Pros:** Proves the full value prop immediately, you'll know if it's useful day 1
- **Cons:** More moving parts to wire up, needs API key setup
- **Completeness:** 8/10

### Approach C: AI-Native Reader
- **Summary:** Skip the traditional feed UI entirely. Just the onboarding + a single daily screen: Claude reads all sources and produces a personalized 5-bullet brief + skill gaps. No feed browsing at all.
- **Effort:** M (human: ~2 weeks / CC: ~2 hours)
- **Risk:** Medium-High (heavy Claude API dependency, hard to debug quality)
- **Pros:** Most opinionated and differentiated
- **Cons:** No feed to browse means less engagement, harder to verify AI quality
- **Completeness:** 7/10

## Recommended Approach

**Approach B: Thin Vertical Slice** — because it proves the full value chain end-to-end while keeping scope manageable. You'll immediately know if this tool is useful to you.

### Accessibility Basics
- **Touch targets:** Minimum 48x48dp for all tappable elements. Chips: 48dp height minimum.
- **Color contrast:** Anduin blue `#3FA6EE` used only for backgrounds/badges, never as text on white (fails WCAG AA at 2.9:1). For text links, use Primary Dark `#2B7AB8` which passes at 4.6:1.
- **Semantic labels:** Add `Semantics` widgets to: skill gap items ("Structured Outputs, rising trend, 12 mentions"), feed articles ("Article from r/MachineLearning: [title]"), bottom nav items.
- **Screen reader:** Brief card reads as continuous paragraph. Gap list reads each item with trend and action.

### Data Flow

1. **Scheduled Cloud Function** (runs every 6 hours): fetches RSS feeds → parses → stores articles in Firestore `articles` collection.
2. **Daily Cloud Function** (runs once at 6 AM UTC): batches recent articles (last 24h) → calls Claude API with each user's profile → writes `DailyBrief` to Firestore → tags matching articles with `isSkillGap: true`.
3. **Flutter app** on launch / pull-to-refresh: reads articles + brief from Firestore (real-time listeners via Riverpod). Offline reads from Firestore's built-in local cache.
4. Display on dashboard; tag matching feed items with SKILL GAP badge.
5. **Weekly Cloud Function** (Sunday 9 AM UTC): computes weekly gap summary per user → sends FCM push notification: "3 new gaps detected this week."

### Cadence

- **Every 6 hours:** Cloud Function fetches RSS feeds and writes new articles to Firestore.
- **Daily (6 AM UTC):** Cloud Function runs Claude API analysis for each user's profile, writes brief to Firestore.
- **Weekly (Sunday 9 AM UTC):** Cloud Function computes weekly gap summary, sends FCM push notification.
- **On app open:** Flutter app reads from Firestore with real-time listeners. Firestore offline cache provides instant reads even without network.

```
  ├── excerpt: "first 200 chars..."
  ├── url: "https://..."
  ├── publishedAt: Timestamp
  ├── tags: ["structured-outputs", "production"]
  ├── isSkillGap: true           // set by Claude analysis
  └── fetchedAt: Timestamp

users/{uid}/briefs/{date}        // date = "2026-03-21"
  ├── summary: "3-sentence personalized brief..."
  ├── gapCount: 3
  └── gaps: [
        { name, trend, mentionCount, suggestedAction, detectedAt, weekId }
      ]

users/{uid}/skillGaps/{weekId}   // weekId = "2026-W12"
  └── gaps: [
        { name, mentionCount, suggestedAction, detectedAt }
      ]
```

Dart model classes mirror these Firestore documents using `freezed` + `json_serializable` for type safety and serialization.

### Trend Indicator Definitions

Trend labels are assigned **client-side in Dart**, not by Claude. Claude only returns skill names and mention counts. The app computes trends using stored historical data:
- **New** — skill not present in any prior week's `SkillGap` records
- **Rising** — skill's `mentionCount` is 2x+ vs. the same skill's count in the previous `weekId`
- **Hot** — skill has 10+ mentions in the current batch (regardless of history)

`weekId` uses ISO 8601 week numbering (Monday = start of week), e.g., `"2026-W12"`.

### Failure States

| Scenario | Behavior |
|----------|----------|
| Claude API timeout/error | Show cached last brief with "Last updated: {date}" label. Feed still works. |
| RSS feed parse error | Skip the failing source, show partial results. Display subtle "1 source unavailable" notice. |
| No network (offline) | Firestore offline cache serves cached articles + brief. Pull-to-refresh shows "No connection" toast. |
| Claude API error (Cloud Function) | Cloud Function logs error, retries once. User sees last cached brief with "Last updated: {date}" label. |
| Reddit RSS rate-limited (429) | Both `.rss` and `.json` endpoints share Reddit's rate limit. Primary mitigation: cache results and limit fetch frequency to once per hour max. Set a descriptive User-Agent header on all requests. |

### Claude API Prompt Design (MVP)

```
System: You are an AI skill analyst. Given a user's role, current skills,
and a batch of recent AI/tech articles, produce a JSON response with:
1. A 3-sentence personalized brief of what's most relevant today
2. A ranked list of skill gaps (skills trending in articles that the
   user doesn't have)
3. For each gap: a trend label and one-line actionable learning suggestion

Trend labels:
- "New" = skill appeared for the first time this week
- "Rising" = mentions increased 2x+ vs. last week
- "Hot" = 10+ mentions in the current batch

Respond ONLY with valid JSON in this exact format:
{
  "brief": "Three sentences about what matters today...",
  "gaps": [
    {
      "skill": "Structured Outputs",
      "mentions": 12,
      "action": "Try the Anthropic cookbook tutorial"
    }
  ],
  "tagged_articles": ["article_id_1", "article_id_2"]
}

Note: "trend" labels are NOT in the API response — they are computed
client-side using historical SkillGap data stored in Firestore.

User profile:
- Role: {role}
- Current skills: {skills}
- Experience: {level}

Recent articles (last 24h):
{article_id}: {title} — {first 200 chars of excerpt}
...
```

**Sample response:**
```json
{
  "brief": "Claude 4.6 dropped with native tool use — relevant to your backend work. Structured outputs are now the default pattern for API integrations. Meanwhile, Cursor adopted agentic workflows that mirror what you're building with WISC.",
  "gaps": [
    {"skill": "Structured Outputs", "mentions": 12, "action": "Try the Anthropic cookbook tutorial"},
    {"skill": "Agentic Tool Use", "mentions": 28, "action": "Build a simple agent with Claude SDK"},
    {"skill": "MCP Servers", "mentions": 9, "action": "Set up your first MCP integration"}
  ],
  "tagged_articles": ["abc123", "def456", "ghi789"]

```

### API Cost Estimate

- Input: ~100 articles × ~100 tokens (title + 200-char excerpt) = ~10K input tokens per call
- Output: ~500 tokens (brief + gaps JSON)
- At Claude Sonnet pricing (~$3/M input, ~$15/M output): **~$0.04/day**, ~$1.20/month
- Mitigation: Cloud Function runs once per day (not on every app open), truncate articles to title + first 200 chars, use Sonnet (not Opus) for extraction
- Firebase costs: Firestore free tier covers 50K reads/20K writes per day — more than sufficient for solo use
